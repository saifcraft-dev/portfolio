A. Missing / wrong environment variables on Vercel (Firebase, etc.)
Works locally but not on Vercel? Extremely common.

Symptoms

White screen
Console shows something like:
FirebaseError: Firebase: No Firebase App '[DEFAULT]' has been created…
ReferenceError: import.meta is not defined
undefined is not an object (evaluating 'firebaseConfig.apiKey')
Checklist

In Vite, env vars must be prefixed with VITE_:

Copy// firebase.ts
const firebaseConfig = {
  apiKey: import.meta.env.VITE_FIREBASE_API_KEY,
  authDomain: import.meta.env.VITE_FIREBASE_AUTH_DOMAIN,
  projectId: import.meta.env.VITE_FIREBASE_PROJECT_ID,
  storageBucket: import.meta.env.VITE_FIREBASE_STORAGE_BUCKET,
  messagingSenderId: import.meta.env.VITE_FIREBASE_MESSAGING_SENDER_ID,
  appId: import.meta.env.VITE_FIREBASE_APP_ID,
};
On Vercel Dashboard → Project → Settings → Environment Variables

Make sure all these exist (names exactly match):

VITE_FIREBASE_API_KEY
VITE_FIREBASE_AUTH_DOMAIN
VITE_FIREBASE_PROJECT_ID
VITE_FIREBASE_STORAGE_BUCKET
VITE_FIREBASE_MESSAGING_SENDER_ID
VITE_FIREBASE_APP_ID
Any other VITE_... you read inside your code (Cloudinary config, etc.)
After editing env vars, trigger a new deployment (variables don’t update on an already built deployment).

B. Protected route / admin guard crashing
If /admin is protected via Firebase Auth and you have some logic like:

Copyconst user = auth.currentUser;
if (!user || !user.isAdmin) {
  // redirect or something
}
or using a hook:

Copyconst { user, loading } = useAuth();
if (!loading && !user) {
  navigate('/login');
}
Common crash patterns:

Accessing user.customClaims or user.role when user is null
Using navigate before BrowserRouter is rendered
Calling hooks conditionally
What to check:

Is there any admin-only logic that assumes user is never null?

Copy// bad
const isAdmin = user.role === 'admin'; // crash if user is null

// safe
const isAdmin = user?.role === 'admin';
If using onAuthStateChanged, ensure you render some fallback while loading:

Copyif (loading) return <div>Loading...</div>;
Send me your AdminRoute (or whatever component wraps /admin) and I can hard‑review it.

C. React Router + Vercel config (less likely, but worth checking)
With Vite + React Router on Vercel, you typically want all paths to serve index.html and let React Router handle routing.

If you’re using Vite’s default build output and Vercel’s Framework Preset = Vite, this is usually handled automatically.

But if you added any custom vercel.json or rewrites/redirects, and /admin is returning a blank document or 404, React won’t boot.

Check:

Go to View Source on /admin
If it has your normal <div id="root"></div> and the script tags, routing is fine and the issue is JS crash, not Vercel routing.
If the HTML is extremely minimal or different from /, share the source or your vercel.json.
D. Direct window / document access at module level
Works in dev but fails in production sometimes, depending on bundling and SSR assumptions.

Examples:

Copyconst token = localStorage.getItem('token'); // at top level
const width = window.innerWidth;            // at top level
These will crash if executed in any environment where window doesn’t exist yet.

Always access them inside effects or with guards:

CopyuseEffect(() => {
  const token = localStorage.getItem('token');
}, []);
or

Copyconst isBrowser = typeof window !== 'undefined';
If you have any such code only in admin-related files, /admin may be the only route that breaks.

E. Build-only type / import issues
Sometimes admin-only components import something that doesn’t exist in the build:

Wrong import path (case sensitivity difference between macOS dev and Linux build on Vercel)
Importing a file that was moved/renamed
On Vercel this often surfaces as:

500 error on a specific chunk in Network tab
Console errors like Uncaught (in promise) Error: Failed to fetch dynamically imported module
Check Network tab → look for admin...js chunks that 404 or 500.
