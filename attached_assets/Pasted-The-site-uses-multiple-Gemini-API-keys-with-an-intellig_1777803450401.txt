The site uses multiple Gemini API keys with an intelligent fallback system to handle rate-limiting and regional restrictions. Here's how:

The System
Configuration (in Replit Secrets):

GEMINI_API_KEY         (Primary)
GEMINI_API_KEY_B       (Fallback 1)
GEMINI_API_KEY_C       (Fallback 2)

How It Works (4-Step Process)
1️⃣ Collect All Keys
const keys = [GEMINI_API_KEY, GEMINI_API_KEY_B, GEMINI_API_KEY_C];

2️⃣ Try Each Key + Model Combination
For each request, the system tries:

Key #1 with gemini-2.5-flash
Key #1 with gemini-2.5-flash-lite (if rate-limited)
Key #1 with gemini-2.0-flash (if still rate-limited)
Key #2 with all 3 models (if Key #1 completely fails)
Key #3 with all 3 models (if Key #2 completely fails)
3️⃣ Return on First Success
As soon as ANY key+model combination succeeds, it returns the response immediately.

4️⃣ Stop on Non-Recoverable Error
If a key returns a hard error (like 400 Bad Request), it stops trying and reports the error immediately instead of wasting time on other keys.

Why Multiple Keys?
Regional Rate-Limiting Problem:
From Pakistan, individual keys frequently get:

429 (Rate-limited)
401/403 (Permission denied)
Even if they work fine elsewhere
The Solution:

Each key has its own quota
3 keys = 3x the request capacity
If Key #1 hits its limit, Key #2 picks up the request
Most users never see a rate-limit error
What Errors Trigger Fallback
Status	Action
429 (Rate-limit)	Try next key ✅
503 (Service busy)	Try next key ✅
401/403 (Auth denied)	Try next key ✅
500/502/504 (Server error)	Try next key ✅
400 (Bad request)	Stop immediately ❌
200 (Success)	Return immediately ✅
Real Example
Request comes in → Key #1 returns 429 (rate-limited) → System automatically tries Key #2 → Key #2 returns 200 (success) → User gets response without delay.

Location in code:

api/_gemini.ts — Shared fallback logic (used by AI and chat endpoints)
api/ai.ts — Uses this fallback system
vite.config.ts — Also contains the same fallback logic for dev mode
